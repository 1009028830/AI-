# 上位机/嵌入式软件开发通用规范

为便于后续统一查看、调试、演示、维护、扩展以及界面布局调整，后续开发过程中应尽量遵循以下通用原则。

## 一、总体原则

开发时应优先考虑以下目标：

- 功能结构清晰
- 界面展示直观
- 模块职责明确
- 代码便于维护
- 功能便于扩展
- 问题便于排查
- 后续便于统一管理

整体上，应尽量形成“**界面清晰、模块独立、逻辑分层、注释规范、中文友好、易于扩展**”的开发风格。

## 二、功能设计原则

### 1. 适合展示和操作的功能，优先在 UI 中体现

对于适合展示、操作、调试和演示的功能，应尽量在 UI 界面中提供可见入口或展示区域，不应无必要地隐藏在内部逻辑中。

界面可体现的内容包括但不限于：

- 功能按钮
- 开关控制
- 菜单入口
- 参数输入框
- 状态指示
- 数据显示区域
- 日志输出区域
- 调试信息区域
- 结果展示区域

目的如下：

- 方便统一查看
- 方便集中调试
- 方便现场演示
- 方便后续调整布局
- 方便统一管理功能入口与状态显示

### 2. 不适合界面化的功能，也必须提前规划

并非所有功能都必须强行做成 UI。
对于底层驱动、后台服务、通信处理、协议解析、自动流程、定时任务、数据处理、系统守护等不适合直接界面化的部分，也必须提前做好规划。

要求包括：

- 模块边界清晰
- 职责划分明确
- 输入输出接口明确
- 便于后续增加日志和调试入口
- 便于后续接入 UI
- 便于单独测试、替换和扩展

不能因为当前不上 UI，就把功能直接散写在各个文件里，导致后续难以维护。

## 三、模块拆分原则

### 1. 新增功能尽量独立拆分

每新增一个功能时，应尽量将其对应实现独立拆分成单独文件、类或模块，不要把多个功能杂糅在同一个源文件中。

原则上做到：

- 一个功能尽量对应一个独立模块
- 一个模块尽量只负责一类事情
- 不同功能之间边界清晰
- 便于单独修改、替换、复用和排查

### 2. 禁止功能堆叠式开发

应尽量避免以下情况：

- 多个功能写在同一个 cpp 文件中
- 多种业务逻辑塞进同一个类中
- 一个界面文件里直接堆大量业务代码
- 一处改动牵连多个不相关功能

### 3. 模块拆分应考虑后续扩展

模块划分不能只满足当前可用，还应考虑：

- 后续是否会增加类似功能
- 后续是否需要替换实现方式
- 后续是否需要单独关闭、启用或调试某模块
- 后续是否需要在 UI 中增加独立入口和状态显示

## 四、分层设计原则

代码应尽量按层次解耦，避免界面、逻辑、数据处理混在一起。

### 1. UI 层

UI 层只负责：

- 界面展示
- 布局组织
- 用户交互
- 事件响应
- 数据绑定与显示刷新

UI 层不应承担复杂业务逻辑，不应直接堆放大量协议处理、通信处理或控制流程。

### 2. 功能逻辑层

功能逻辑层负责：

- 具体功能实现
- 状态流转
- 业务流程控制
- 功能接口对外提供

该层应尽量独立于具体界面，保证即使更换界面，也不影响功能本身。

### 3. 数据处理层/服务层

数据处理层负责：

- 数据解析
- 数据转换
- 协议封装与解包
- 设备通信
- 文件处理
- 日志处理
- 后台服务调度

这部分应尽量与 UI 解耦，不直接依赖具体界面控件。

## 五、界面开发原则

### 1. 界面以“可见、可调、可查”为主

凡适合界面化的功能，设计时应优先考虑：

- 是否有明确入口
- 是否能看到当前状态
- 是否能修改关键参数
- 是否能查看结果和日志
- 是否便于调试和演示

### 2. 界面布局应便于后续调整

界面开发时应尽量避免把布局和逻辑强耦合，避免后续改布局时需要大范围修改功能代码。

要求：

- 控件布局尽量独立管理
- 交互逻辑与界面展示分离
- 状态更新尽量通过统一接口完成
- 为后续重构界面预留空间

### 3. 状态展示应尽量明确

界面中的状态信息应尽量做到直观、明确、可理解，例如：

- 已连接 / 未连接
- 运行中 / 已停止
- 成功 / 失败
- 当前参数值
- 当前任务进度
- 最近一次执行结果

避免只有内部变量变化而界面无任何体现。

## 六、代码结构原则

### 1. 命名要明确

命名应尽量做到见名知义，不使用含义模糊、范围过大的名称。

要求：

- 类名明确
- 文件名明确
- 变量名明确
- 函数名明确
- 模块名明确

### 2. 职责要单一

每个文件、类、模块都应尽量只负责一类事情，避免职责混乱。

### 3. 目录结构要清晰

工程目录应尽量分层管理，例如可按以下思路划分：

- `ui/`：界面相关
- `modules/`：功能模块
- `services/`：服务或后台处理
- `data/`：数据处理
- `protocol/`：协议相关
- `utils/`：公共工具
- `logs/`：日志相关

目录划分不要求死板一致，但应保持长期统一、含义清晰。

## 七、注释规范

代码应根据实际需要添加清晰、准确、可维护的注释。

### 1. 必须注释的内容

以下内容应尽量补充说明：

- 核心功能作用
- 关键业务流程
- 重要参数含义
- 单位与取值范围
- 状态值说明
- 协议字段含义
- 关键接口输入输出
- 特殊处理逻辑和注意事项

### 2. 注释要求

注释应满足：

- 内容准确
- 说明清楚
- 与代码保持一致
- 避免无意义注释
- 避免注释缺失导致难以维护

### 3. 修改代码时同步更新注释

后续修改或扩展功能时，应同步检查并更新注释，避免出现“代码已变，注释未改”的情况。

## 八、中文优先原则

为便于统一理解、管理、调试、演示和维护，标题、界面文字、注释及说明内容应尽量使用中文表达。

适用范围包括：

- 页面标题
- 分组标题
- 按钮名称
- 参数说明
- 状态说明
- 日志标题
- 功能描述
- 注释说明
- 文档标题

对于必须保留英文的内容，如：

- 第三方库接口名
- 协议字段名
- 标准术语
- 系统函数名
- 文件格式关键字

可保留英文，但应尽量配套中文说明。

## 九、调试与维护原则

开发时应尽量考虑后期调试和维护便利性。

要求包括：

- 功能入口清晰
- 状态输出明确
- 错误信息可追踪
- 日志内容有意义
- 模块可单独排查
- 问题可快速定位

对于关键功能，最好预留：

- 状态显示
- 日志输出
- 参数查看
- 调试入口
- 错误提示

## 十、扩展性原则

所有设计应尽量为后续扩展预留空间，不做只满足当前一次性需求的混乱实现。

应尽量保证：

- 新增功能不破坏原有结构
- 替换一个功能不影响其他模块
- 修改 UI 不影响核心逻辑
- 修改逻辑不强依赖具体布局
- 后续新增页面、设备、协议、数据项时，能较平滑接入

## 十一、推荐执行要求

后续开发默认尽量按以下方式执行：

1. 先规划功能属于 UI 展示、逻辑实现还是数据处理
2. 再确定是否需要独立模块
3. 每个新增功能尽量单独建文件或模块
4. 界面交互与功能实现尽量分离
5. 状态、日志、参数、结果尽量可见
6. 注释、标题、界面文字尽量中文化
7. 代码结构始终保持清晰可维护

## 十二、统一开发目标

后续开发应尽量达到以下目标：

- 能展示的功能尽量展示到 UI
- 不适合展示的功能也要提前规划结构
- 每个功能尽量独立拆分
- UI、逻辑、数据处理尽量分层
- 标题、注释、说明尽量中文
- 代码清晰、模块单一、便于维护
- 方便后续统一查看、调试、演示、扩展和排查

---

## 固定调用语句

后续可直接使用以下要求调用本规范：

> 按通用开发规范执行：功能优先界面可见，模块独立拆分，界面与逻辑分离，代码结构清晰，注释规范，标题和说明尽量使用中文，不适合 UI 的功能也要提前做好模块规划。
