# 代码开发规范（方案审核流程版）

## 1. 文档目的

本规范用于约束后续所有代码开发、功能新增、项目重构、问题修复和模块扩展流程。

核心目标是：

- 开发前先明确思路
- 开发前先形成 MD 方案文档
- 方案交由用户审核
- 审核通过后再开始写代码
- 代码实现保持最小改动
- 代码结构保持清晰
- 严格按照已确认方案执行
- 不擅自增加未确认功能
- 不进行无依据猜测
- 开发过程中记录问题、原因和解决方法
- 形成长期问题记忆文件，避免重复犯错

本规范适用于 Qt、C++、Python、ROS、CAN、Modbus、WebSocket、嵌入式 Linux、Buildroot、控制器程序、诊断程序、后端服务、命令行工具等后续项目开发。

---

## 2. 总体开发原则

### 2.1 先写方案，再写代码

任何功能开发或代码修改，不允许直接开始写代码。

必须先完成：

1. 梳理需求
2. 分析现有结构
3. 给出实现思路
4. 编写 MD 方案文档
5. 交由用户审核
6. 审核通过后再进入编码阶段

### 2.2 严格按审核通过的方案执行

代码开发必须以用户确认后的 MD 方案为准。

禁止在编码过程中擅自增加：

- 新功能
- 新界面
- 新逻辑
- 新通信协议
- 新数据结构
- 新依赖库
- 新配置项

如确实发现必须调整，应先停止编码，重新说明原因，并更新 MD 方案后再交由用户审核。

### 2.3 最小代码量实现

实现功能时优先采用最小代码量完成需求。

要求：

- 能复用已有代码就不要重复写
- 能简单实现就不要复杂封装
- 能局部修改就不要大范围重构
- 能不引入新依赖就不引入新依赖
- 能不改公共接口就不改公共接口

最小代码量不是随便写，而是在保证结构清晰、职责明确、功能正确的前提下，减少不必要代码。

### 2.4 结构清晰优先

代码必须保持清晰、可读、可维护。

要求：

- 文件职责明确
- 模块职责单一
- 命名直观
- 调用关系简单
- 不堆积无关逻辑
- 不写万能类
- 不把所有代码塞进一个文件

### 2.5 不多加猜测

开发时只做用户明确要求的内容。

禁止因为“可能以后会用到”而提前增加大量未确认功能。

允许保留合理扩展接口，但不能影响当前功能的简洁性和可读性。

## 3. 标准开发流程

后续所有代码相关任务，统一按照以下流程执行。

```text
需求输入
  ↓
需求理解与边界确认
  ↓
读取长期问题记录文件
  ↓
编写开发思路
  ↓
编写 MD 方案文档
  ↓
用户审核
  ↓
用户确认通过
  ↓
开始写代码
  ↓
开发过程中记录问题
  ↓
自查代码结构
  ↓
更新长期问题记录文件
  ↓
给出修改说明与测试方法
```

---

## 4. 第一步：需求理解

在写方案前，必须先明确需求。

### 4.1 需要明确的内容

- 要实现什么功能
- 修改哪个模块
- 是否涉及 UI
- 是否为纯后端程序
- 是否涉及通信
- 是否涉及数据保存
- 是否涉及配置参数
- 是否影响原有功能
- 是否需要兼容现有接口
- 是否需要提供测试方法

### 4.2 不明确时的处理方式

如果需求中存在关键不明确点，应先提出问题。

但如果问题不影响整体方案，可以先按最小合理假设写出方案，并在 MD 中明确标注：

```text
当前假设：xxx
待确认点：xxx
```

禁止在代码中直接按猜测实现。

---

## 5. 第二步：开发前读取长期问题记录文件

每次开始新的代码开发、功能修改、问题修复或项目重构前，应优先读取长期问题记录文件。

推荐文件路径：

```text
docs/development_memory.md
```

读取重点：

- 以前是否遇到过类似问题
- 是否有已经验证过的解决方法
- 是否有禁止修改或容易出错的文件
- 是否有特殊环境依赖
- 是否有编译、运行、部署注意事项
- 是否有历史遗留问题
- 是否有之前踩过的坑

如果没有该文件，应在本次开发中创建。

---

## 6. 第三步：编写开发思路

在 MD 文档正式方案前，应先给出简短开发思路。

开发思路应包括：

- 当前要解决的问题
- 准备采用的实现方式
- 需要新增或修改的文件
- 是否影响现有逻辑
- 是否涉及 UI
- 如果不涉及 UI，采用什么方式展示状态
- 是否需要配置项
- 是否需要测试验证

示例：

```text
本次功能准备采用最小改动方式实现。
该程序属于后端通信服务，不新增 UI。
运行状态通过日志输出，启动参数通过配置文件管理。
具体通信逻辑放在 comm 层，业务流程放在 logic 层，数据保存放在 data/storage 中。
```

---

## 7. 第四步：编写 MD 方案文档

所有代码开发前，必须先生成 MD 方案文档。

建议文档放在：

```text
docs/
```

推荐命名方式：

```text
docs/功能名称_开发方案.md
docs/模块名称_重构方案.md
docs/问题名称_修复方案.md
```

例如：

```text
docs/485从机功能开发方案.md
docs/CAN调试页面重构方案.md
docs/数据保存功能开发方案.md
docs/WebSocket通信服务开发方案.md
```

---

后续每个功能开发方案，建议使用以下模板。

```markdown
# xxx 功能开发方案

说明本次要实现的功能，以及用户提出的核心要求。

## 2. 当前现状

说明当前项目中已有的相关代码、模块、文件或功能。

## 3. 历史问题参考

本次开发前已查看 docs/development_memory.md。

相关历史问题：

- xxx

本次需要避免：

- xxx

如果没有相关问题，写：

当前未发现与本次功能直接相关的历史问题。

## 4. 实现目标

列出本次开发完成后应达到的目标。

- 最小代码量实现
- 不影响现有功能
- 结构清晰
- 严格按本方案执行
- 不增加未确认功能
- 不进行无依据猜测

## 6. 项目类型与 UI 适配说明

本次项目类型：

- Qt 界面程序 / 后端服务程序 / ROS 节点 / 命令行工具 / 通信桥接程序

是否涉及 UI：

- 是 / 否

处理方式：

- 如果涉及 UI：说明新增哪些按钮、状态、参数、日志显示
- 如果不涉及 UI：说明通过日志、配置文件、命令行参数或状态接口体现运行状态

### 7.1 新增文件

```text
xxx/xxx.h
xxx/xxx.cpp
```

### 7.2 修改文件

```text
xxx/xxx.cpp
xxx/xxx.ui
```

### 7.3 不修改文件

```text
xxx/xxx.cpp
```

| 模块 | 文件 | 职责 |
|---|---|---|
| UI层 | xxx | 负责界面显示和用户操作 |
| Service层 | xxx | 负责服务启动和生命周期 |
| Logic层 | xxx | 负责业务调度 |
| Comm层 | xxx | 负责通信收发 |
| Data层 | xxx | 负责数据解析或保存 |
| Config层 | xxx | 负责配置加载和保存 |

## 9. 实现流程

```text
用户操作 / 程序启动
  ↓
UI 层或 Service 层接收入口
  ↓
Logic 层处理流程
  ↓
Comm/Data/Config 层执行具体动作
  ↓
返回结果给 Logic 层
  ↓
UI 显示或日志输出运行状态
```

说明核心处理逻辑，但不写具体代码。

## 11. 配置项说明

说明是否新增配置参数、默认值、保存位置。

## 12. 数据保存说明

说明是否需要保存数据、保存格式、保存路径、字段内容。

## 13. 错误处理说明

说明连接失败、参数错误、通信异常、文件写入失败等情况如何处理。

## 14. 测试方法

列出编译、运行、功能测试、异常测试方法。

## 15. 预计影响范围

说明本次修改是否影响原有功能。

## 16. 待用户确认事项

列出需要用户审核确认的内容。
```

---

## 9. 用户审核要求

MD 方案写完后，必须交由用户审核。

审核重点包括：

- 功能是否符合需求
- 文件拆分是否合理
- 是否有多余功能
- 是否有未确认的猜测
- 是否需要减少代码量
- 是否影响现有功能
- 是否需要 UI
- 后端程序是否保持简单

在用户没有明确确认之前，不进入写代码阶段。

---

## 10. 编码执行规范

用户确认 MD 方案后，才开始写代码。

### 10.1 编码要求

- 严格按照 MD 文件执行
- 不擅自增加方案外内容
- 不随意更改目录结构
- 不破坏现有功能
- 不引入不必要依赖
- 不写无用封装
- 不写重复逻辑

### 10.2 修改范围控制

只修改方案中列出的文件。

如果发现需要修改方案外文件，应先说明原因，更新方案后再继续。

### 10.3 最小改动原则

优先局部修改。

例如：

- 只需要增加一个状态输出，就不要重写整个程序
- 只需要增加一个解析函数，就不要重构整个协议模块
- 只需要保存一类数据，就不要设计复杂数据库系统
- 只需要读取一个配置项，就不要引入大型配置框架

---

## 11. 代码分层规范

代码按照项目类型进行分层。

### 11.1 通用推荐结构

```text
project_root/
├── docs/      // 文档、方案、问题记录
├── logic/     // 业务逻辑层
├── comm/      // 通信层
├── data/      // 数据层
├── config/    // 配置层
├── utils/     // 工具层
├── tests/     // 测试
└── main.cpp   // 程序入口
```

### 11.2 带 UI 项目推荐结构

```text
project_root/
├── docs/
├── ui/        // 界面层
├── logic/
├── comm/
├── data/
├── config/
├── utils/
├── tests/
└── main.cpp
```

### 11.3 纯后端程序推荐结构

```text
project_root/
├── docs/
├── service/   // 服务启动、主流程、生命周期管理
├── logic/
├── comm/
├── data/
├── config/
├── utils/
├── tests/
└── main.cpp
```

### 11.4 各层职责

#### UI 层

仅用于有界面的项目。

职责：

- 页面显示
- 用户操作
- 状态展示
- 日志展示
- 调用 logic 层接口

禁止：

- 直接写复杂业务逻辑
- 直接写底层通信
- 直接写协议解析
- 直接写大量文件保存逻辑

#### Service 层

用于后端服务、命令行程序、systemd 服务、ROS 节点等。

职责：

- 程序启动入口
- 参数解析
- 服务生命周期管理
- 初始化各模块
- 捕获退出信号
- 输出运行状态

#### Logic 层

职责：

- 业务流程控制
- 功能调度
- 状态判断
- 调用 comm/data/config
- 向 UI 或 Service 提供接口

#### Comm 层

职责：

- CAN 通信
- 串口通信
- TCP/UDP 通信
- WebSocket 通信
- Modbus 通信
- 连接状态管理

#### Data 层

职责：

- 数据模型
- 数据解析
- 数据转换
- 数据缓存
- 数据保存

#### Config 层

职责：

- 默认参数
- 配置加载
- 配置保存
- 运行参数管理

#### Utils 层

职责：

- 时间工具
- 文件工具
- 字符串工具
- 日志辅助
- 通用函数

注意：具体业务逻辑不要放到 utils 中。

## 12. UI 与后端程序适配规范

### 12.1 不强制所有项目都有 UI

后续开发中，并不是所有程序都必须包含 UI。

项目可能分为：

- Qt 图形界面程序
- Web 管理界面程序
- 后端服务程序
- ROS 节点程序
- 通信桥接程序
- 命令行工具程序
- 守护进程 / systemd 服务程序
- 数据处理脚本

原则是：

```text
有 UI 的项目，尽量把功能入口、状态、参数、日志在界面中体现。
没有 UI 的后端程序，不为了展示而强行增加 UI。
```

### 12.2 有 UI 项目的要求

如果项目本身是界面程序，或者用户明确要求带 UI，则应遵守 UI 可视化原则。

应尽量展示：

- 连接状态
- 运行状态
- 参数配置
- 收发计数
- 当前模式
- 错误提示
- 日志输出
- 数据展示

UI 文件只负责：

- 显示
- 交互
- 调用接口
- 更新界面

复杂逻辑必须放到 logic 层。

推荐页面按以下区域组织：

```text
功能操作区
参数配置区
状态显示区
数据展示区
日志调试区
```

如果使用 Qt 项目：

- `.ui` 文件负责界面布局
- `.h/.cpp` 文件负责控件绑定和信号槽
- 不要把核心业务逻辑写进页面 cpp 中

### 12.3 纯后端程序的要求

如果项目是纯后端程序、命令行程序、ROS 节点、通信服务或 systemd 守护进程，则不强制增加 UI。

此类程序重点要求：

- 启动参数清晰
- 配置文件清晰
- 日志输出清晰
- 状态可查询
- 错误提示明确
- 模块职责单一
- 方便部署和排查问题

### 12.4 后端程序的可观测性要求

纯后端程序虽然不要求 UI，但必须保证可观测、可调试。

至少应提供以下能力中的一种或多种：

- 控制台日志
- 文件日志
- 状态打印
- 命令行参数
- 配置文件
- 运行状态接口
- systemd service 状态
- ROS topic / service / param
- WebSocket / TCP 状态反馈

例如：

```bash
./can_web_bridge can0 ws://127.0.0.1:9000 3000
```

应能从日志中看出：

- 当前使用的 CAN 口
- 当前连接的服务器地址
- 是否连接成功
- 是否正在重连
- 已发送数量
- 已接收数量
- 最后一次错误原因

### 12.5 判断是否需要 UI

是否需要 UI，以用户需求和项目类型为准。

判断规则：

```text
用户明确要求 UI      -> 做 UI
项目本身已有 UI      -> 在 UI 中体现
需要演示和人工操作  -> 优先考虑 UI
只是后台服务        -> 不强制 UI
只是通信桥接        -> 优先日志和配置
只是 ROS 节点       -> 优先参数、topic、日志
只是一次性脚本      -> 优先命令行参数和输出
```

## 13. 文件拆分规范

### 13.1 一个功能尽量独立成模块

每新增一个功能，尽量独立文件实现。

带 UI 项目示例：

```text
ui/pages/modbus_slave_page.h
ui/pages/modbus_slave_page.cpp
logic/modbus/modbus_slave_controller.h
logic/modbus/modbus_slave_controller.cpp
comm/modbus/modbus_slave_service.h
comm/modbus/modbus_slave_service.cpp
data/storage/modbus_data_storage.h
data/storage/modbus_data_storage.cpp
```

纯后端项目示例：

```text
service/app_service.h
service/app_service.cpp
logic/bridge/can_web_bridge_controller.h
logic/bridge/can_web_bridge_controller.cpp
comm/can/can_driver.h
comm/can/can_driver.cpp
comm/websocket/websocket_client.h
comm/websocket/websocket_client.cpp
config/app_config.h
config/app_config.cpp
utils/log_helper.h
utils/log_helper.cpp
```

### 13.2 禁止万能文件

禁止一个文件同时负责：

- UI 显示
- 通信收发
- 协议解析
- 数据保存
- 配置读取
- 日志处理

### 13.3 文件命名要求

文件名必须能看出作用。

推荐命名：

```text
can_driver.h
can_parser.h
can_debug_controller.h
can_record_storage.h
serial_port_driver.h
modbus_slave_service.h
config_manager.h
log_helper.h
```

不推荐命名：

```text
test.cpp
new.cpp
fun.cpp
tools.cpp
data.cpp
main2.cpp
```

## 14. 问题记录与长期记忆文件规范

### 14.1 必须建立长期问题记录文件

后续项目开发过程中，必须建立一个长期维护的问题记录文件。

推荐文件路径：

```text
docs/development_memory.md
```

该文件用于记录开发、编译、运行、调试、部署过程中遇到的问题，形成项目长期经验库。

目的：

- 避免重复犯同样错误
- 方便后续快速排查问题
- 方便项目交接和维护
- 记录真实开发过程中的坑点
- 固化已经验证过的解决方法

### 14.2 每次开发前必须读取

每次开始新的代码开发、功能修改、问题修复或项目重构前，应优先查看：

```text
docs/development_memory.md
```

重点查看：

- 以前是否遇到过类似问题
- 是否有已经验证过的解决方法
- 是否有禁止修改或容易出错的文件
- 是否有特殊环境依赖
- 是否有编译、运行、部署注意事项

读取后再编写新的 MD 开发方案，尽量避免重复踩坑。

### 14.3 开发过程中随时更新

开发过程中只要遇到以下情况，就应及时记录到长期问题文件中：

- 编译错误
- 链接错误
- 运行异常
- 通信失败
- UI 显示异常
- 配置读取失败
- 数据保存失败
- 依赖库缺失
- 环境变量问题
- 权限问题
- 路径问题
- 编码格式问题
- 设备连接问题
- 调试过程中发现的隐藏问题

不要等项目结束后再凭记忆补写，应该在问题解决后及时记录。

## 18. 禁止事项

后续开发中必须避免以下问题。

### 18.1 禁止未审核直接写代码

没有 MD 方案、没有用户确认，不直接写代码。

### 18.2 禁止擅自扩展功能

用户没要求的功能，不主动加。

### 18.3 禁止大范围无关重构

只改本次需求相关内容。

### 18.4 禁止 UI 文件堆积逻辑

UI 层不能堆通信、解析、保存、业务判断。

### 18.5 禁止强制后端程序增加 UI

纯后端服务、通信桥接、ROS 节点、命令行工具，不为了展示而强行增加 UI。

### 18.6 禁止复杂化简单功能

简单功能简单实现，不为了“看起来高级”增加复杂结构。

### 18.7 禁止引入不必要依赖

除非方案中明确说明，否则不新增第三方库。

### 18.8 禁止破坏现有功能

修改前必须考虑原有功能是否受影响。

## 19. 长期执行规则

### 规则 1

所有代码开发先写 MD 方案，再写代码。

### 规则 2

MD 方案必须经过用户审核确认。

### 规则 3

代码只按照确认后的方案执行。

### 规则 4

实现逻辑使用最小代码量。

### 规则 5

保持目录结构、模块结构、调用关系清晰。

### 规则 6

不要多加猜测，不做未确认功能。

### 规则 7

有 UI 的项目，UI 层只做显示和交互，复杂逻辑放到 logic 层。

### 规则 8

没有 UI 的后端程序，不强行增加 UI，重点做好日志、配置、状态输出和错误提示。

### 规则 9

通信、数据、配置、工具必须分层管理。

### 规则 10

主要文件、类、函数必须有 Doxygen 注释。

### 规则 11

每次开发前必须读取 docs/development_memory.md。

### 规则 12

开发过程中遇到问题必须记录问题、原因和解决方法。

### 规则 13

每次交付必须包含修改说明、测试方法和问题记录更新情况。

## 20. 结论

后续所有代码开发必须遵循本规范。

核心执行顺序固定为：

```text
先读取历史问题记录
再想清楚实现思路
再写 MD 方案
先给用户审核
审核通过后写代码
最小代码量实现
保持结构清晰
严格按方案执行
开发中记录问题
开发后更新长期问题文件
```

该规范的重点不是让代码变复杂，而是让项目长期保持：

- 好理解
- 好修改
- 好调试
- 好维护
- 好交接
- 不乱改
- 不乱加功能
- 不重复犯错
- 不破坏原有逻辑
