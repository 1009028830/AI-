# 项目重构与开发规范

## 1. 文档目的

为便于后续统一查看、调试、演示、维护以及界面布局调整，在**不影响现有功能**的前提下，对项目进行规范化整理与重构，形成统一、清晰、可长期执行的开发规范。

本规范重点约束以下几个方面：

- 功能可视化
- UI分类清晰
- 代码按层分离
- 文件按类型分类存放
- 功能模块独立
- 代码说明完整
- 注释风格统一为 Doxygen
- AI 编码助手执行流程规范
- Spec / Plan / Build / Test / Review / Ship 全流程质量门禁
- 测试前置与测试即证明
- 薄垂直切片增量实现
- 通信与数据解析安全防御红线
- 反合理化防逃避机制

---

## 2. 整体目标

项目整理与重构后，应达到以下目标：

### 2.1 功能可视化
所有可以展示在界面中的功能，尽量在 UI 中体现，方便后续查看、调试、演示和调整布局。

重点包括但不限于：

- 功能入口
- 参数配置
- 状态显示
- 数据展示
- 调试开关
- 日志输出
- 错误提示

### 2.2 UI分类清晰
界面中的功能应按功能类型进行分类组织，保证结构清晰、层次分明，避免功能杂乱堆叠。

### 2.3 代码按层分离
UI层、业务逻辑层、数据层、通信层、配置层、工具层职责分明，尽量降低耦合。

### 2.4 文件分类统一
不同类型代码按目录归类管理，避免多个不相关功能混杂在同一文件中。

### 2.5 模块独立清晰
每个功能模块职责尽量单一，结构明确，便于后续维护、替换、排查问题和继续扩展。

### 2.6 代码说明完整
主要模块、主要文件、主要类、主要函数均应具备说明，避免后续每次重新梳理代码。

### 2.7 注释风格统一
统一使用 **Doxygen** 风格注释，保证接口说明标准化、可自动生成文档。

---

## 3. 总体重构原则

### 3.1 不影响现有功能
重构过程中应优先保证已有功能可正常运行，不以破坏现有逻辑为代价追求结构美观。

### 3.2 先定边界与测试，再做垂直切片
重构不允许先大面积移动文件、拆目录、改调用关系后再补测试。正确顺序应为：

1. 明确本次只处理的最小业务流。
2. 写出该业务流的验收标准和验证命令。
3. 为 Logic 层、Comm 层、Data 解析层补充最小失败测试。
4. 只围绕该业务流完成从 Data / Comm 到 Logic，再到 UI 或输出端的垂直打通。
5. 测试通过后，再继续下一个业务流。
6. 禁止一次性横向重构多个层级后再统一验证。

### 3.3 一个模块只负责一类事情
每个模块应尽量做到职责单一，例如：

- UI模块：负责展示和交互
- 逻辑模块：负责流程控制和业务处理
- 通信模块：负责收发和连接状态管理
- 数据模块：负责解析、转换、缓存、存储
- 配置模块：负责参数加载、保存、默认值管理
- 工具模块：负责通用公共能力

### 3.4 能展示的尽量展示
凡是后续可能需要演示、调试、维护、观测的问题，都应尽量在界面中留出入口或状态显示。

## 5. UI展示与组织规范

## 5.1 UI总体要求
所有功能尽量都要在 UI 界面中体现出来，方便后续查看、调试、演示以及调整布局。

不要将关键功能入口、状态显示或重要操作隐藏在底层逻辑中。能在界面展示的内容尽量都要体现出来，例如：

- 按钮
- 开关
- 状态指示
- 参数配置区
- 数据显示区
- 日志区
- 调试区

## 5.2 UI页面推荐分区
建议每个页面按以下区域组织：

### 1）功能操作区
用于放置：

- 启动
- 停止
- 连接
- 断开
- 复位
- 清空
- 测试开始/结束

### 2）参数配置区
用于放置：

- 输入框
- 下拉框
- 复选框
- 单选框
- 通信参数
- 模式选择
- 阈值配置

### 3）状态显示区
用于放置：

- 连接状态
- 运行状态
- 故障状态
- 当前模式
- 在线/离线
- 心跳状态
- 收发计数

### 4）数据展示区
用于放置：

- 实时数据
- 表格
- 波形
- 原始报文
- 解析结果
- 关键指标显示

### 5）日志调试区
用于放置：

- 日志输出
- 错误提示
- 调试输出
- 报文收发记录
- 操作记录

## 5.3 UI功能分类要求
UI界面中的功能必须按照功能类型进行分类组织，保证层次清晰。

推荐采用以下方式组织：

```text
主界面
├── 首页总览
├── 设备连接
├── 参数配置
├── 实时监控
├── 通信调试
├── 数据记录
├── 日志查看
├── 系统设置
└── 关于/帮助
```

### UI组织原则

- 每个页面尽量只承载一类核心功能
- 页面内部使用 GroupBox/区域框进行细分
- 正式功能与调试功能应区分显示
- 状态显示与对应操作尽量靠近
- 不允许将所有按钮堆在一个页面中

## 6. 代码结构规范

## 6.1 分层要求
代码必须按照功能类型和职责进行分类管理。

推荐分层如下：

- `ui/`：界面与布局相关代码
- `logic/`：业务逻辑相关代码
- `data/`：数据处理相关代码
- `comm/`：通信相关代码
- `config/`：配置相关代码
- `utils/`：工具类与公共方法代码

## 6.2 各层职责定义

### 6.2.1 UI层
**职责：**

- 页面显示
- 用户交互
- 信号槽连接
- 调用逻辑层接口
- 展示逻辑处理结果

**禁止：**

- 不要直接写复杂业务逻辑
- 不要直接做协议解析
- 不要直接做底层通信细节处理
- 不要把大量流程控制写在 UI 文件里

### 6.2.2 Logic层
**职责：**

- 业务流程控制
- 模块联动
- 状态判断
- 功能调度
- 向 UI 提供统一可调用接口

### 6.2.3 Comm层
**职责：**

- 通信初始化
- 建立连接
- 数据发送
- 数据接收
- 重连机制
- 通信状态管理
- 协议帧收发接口封装

### 6.2.4 Data层
**职责：**

- 数据结构定义
- 报文解析
- 数据转换
- 数据缓存
- 数据持久化

### 6.2.5 Config层
**职责：**

- 配置加载
- 默认参数管理
- 配置保存
- 运行参数持久化

### 6.2.6 Utils层
**职责：**

- 提供通用工具函数
- 提供公共能力
- 提供可复用辅助方法

**要求：**
- 与具体业务强耦合的代码不要放入 utils

## 7. 文件拆分规范

## 7.1 一个功能尽量独立成模块
每新增一个功能，都应尽量独立成单独文件，不要将多个不相关功能混写在同一个文件中。

例如“CAN调试”功能，不应把页面显示、发送逻辑、接收逻辑、协议解析、数据保存全部写进一个文件。

推荐拆分如下：

```text
ui/pages/can_debug_page.h
ui/pages/can_debug_page.cpp

logic/device/can_debug_controller.h
logic/device/can_debug_controller.cpp

comm/can/can_driver.h
comm/can/can_driver.cpp

data/parser/can_parser.h
data/parser/can_parser.cpp

data/storage/can_record_storage.h
data/storage/can_record_storage.cpp
```

## 7.2 文件命名要求
命名必须清晰体现功能与职责，不允许出现大量含义不明确的文件名。

### 推荐命名风格

- UI类：`MainWindow`、`CanDebugPage`、`ConfigPage`
- 逻辑类：`CanDebugController`、`TestFlowManager`
- 通信类：`CanDriver`、`SerialPortDriver`、`ModbusSlaveService`
- 数据类：`CanFrameModel`、`DeviceStatusModel`、`DataParser`
- 配置类：`ConfigManager`、`DeviceConfigLoader`
- 工具类：`LogHelper`、`TimeHelper`、`FileHelper`

---

## 8. 模块调用关系规范

推荐调用依赖如下：

```text
UI层  ->  Logic层  ->  Comm层 / Data层 / Config层
                     ->  Utils层

Comm层 <-> Data层
Logic层 <-> Data层
Config层 -> Logic层 / Comm层 / UI层
Utils层 -> 全局复用
```

### 约束要求

- UI层不能直接依赖底层协议细节
- Comm层不能依赖 UI层
- Data层尽量保持独立
- Utils层不写具体业务流程
- Logic层作为主要调度入口

---

## 9. 代码说明要求

每个主要功能模块、主要文件都需要提供对应的代码介绍。

代码介绍至少应包含以下内容：

### 9.1 文件或模块说明内容

- 文件或模块的作用
- 所属功能类别
- 主要类、函数或接口说明
- 输入输出关系
- 与其他模块的调用关系
- 后续维护或修改时需要注意的重点

### 9.2 文档要求
代码说明应尽量清晰直观，方便后续快速理解整体结构，而不是每次都重新梳理代码。

建议在 `docs/` 目录中建立模块说明文档，例如：

```text
docs/
├── overall_architecture.md
├── ui_design_spec.md
├── comm_module_spec.md
├── logic_module_spec.md
├── data_module_spec.md
├── config_spec.md
└── coding_rule_doxygen.md
```

## 10. Doxygen 注释规范

项目中所有主要文件、类、函数、成员变量、枚举，统一使用 **Doxygen** 风格注释。

 *
 * @note 该类不负责业务逻辑处理，仅提供通信能力。

{
};

```cpp
/**
 * @brief 发送一帧CAN报文

 * @return false 发送失败

## 10.5 成员变量注释模板

```cpp
private:
    /** @brief 当前CAN设备是否已连接 */

    /** @brief 已发送报文计数 */
    uint32_t m_txCount;

## 10.6 枚举注释模板

```cpp
/**
 * @enum DeviceState
 * @brief 设备运行状态枚举
 */
enum class DeviceState
{
    Idle,       /**< 空闲状态 */
    Running,    /**< 运行状态 */
    Error,      /**< 故障状态 */
    Offline     /**< 离线状态 */

```

## 10.7 注释编写要求

- 注释应描述“为什么”和“做什么”，不是简单重复代码字面含义
- 关键流程、状态机、协议字段必须写清楚用途
- 输入输出必须明确
- 容易出错或后期容易修改的部分必须加维护说明
- 禁止完全无注释的核心代码存在

## 11. 测试验证门禁（Testing & Verification Strategy）

本规范将测试视为功能可用、重构安全、AI 输出可信的证明。没有测试记录的修改，不得判定为完成。

### 11.1 测试即证明

任何代码修改必须同时给出验证依据。验证依据可以是自动化测试、编译输出、运行日志、通信报文、UI 操作记录、服务状态或实际设备测试记录。

禁止使用以下表述作为交付结论：

- 应该可以
- 理论上没问题
- 看起来正常
- 代码逻辑比较简单，所以不测
- 后面有时间再补测试

允许的交付结论必须包含：

```text
1. 执行了什么测试
2. 使用了什么命令或操作步骤
3. 期望结果是什么
4. 实际结果是什么
5. 是否通过
6. 未覆盖的风险是什么
```

### 11.2 TDD 红-绿-重构纪律

当修改 Logic 层、Comm 层、Data 解析层、协议字段、状态机、数据模型、配置加载逻辑时，必须优先采用 TDD 流程。

```text
RED    ->  GREEN  ->  REFACTOR
先失败     再通过     再整理结构
```

执行要求如下：

| 阶段 | 必须完成的动作 | 禁止事项 |
|---|---|---|
| RED | 先写出能暴露问题或覆盖新需求的失败测试 | 未写失败测试就直接写生产代码 |
| GREEN | 用最小代码让测试通过 | 一次性重写大段逻辑 |
| REFACTOR | 在测试保护下整理结构、命名、分层和注释 | 重构后不重新运行测试 |

对于无法立即自动化的嵌入式硬件、CAN、串口、ROS、systemd 服务类任务，也必须写出手动测试脚本或验证步骤，作为临时替代。

### 11.3 分层测试要求

不同层级必须具备不同类型的验证方式。

| 层级 | 必须验证内容 | 推荐验证方式 |
|---|---|---|
| UI 层 | 控件入口、状态显示、参数输入、错误提示 | 手动 UI 操作记录 / 截图 / UI 自动化 |
| Logic 层 | 状态机、流程分支、边界条件、异常路径 | 单元测试 / 场景测试 |
| Comm 层 | 连接、断开、重连、收发、超时、异常帧 | mock 设备 / vcan / 串口回环 / 报文日志 |
| Data 层 | 解析、转换、字段范围、非法输入 | 单元测试 / fuzz 样例 / 边界样例 |
| Config 层 | 默认值、加载失败、保存失败、兼容旧配置 | 单元测试 / 配置文件替换测试 |
| Utils 层 | 通用函数输入输出和异常路径 | 单元测试 |
| 系统服务 | 启动、停止、自启动、日志、权限、依赖 | systemctl / journalctl / ps / pidof |

### 11.4 必须保留测试产物

每次功能新增、Bug 修复或重构，至少保留以下测试产物之一：

```text
docs/test_reports/xxx_test_report.md
tests/xxx_test.cpp
tests/xxx_test.py
scripts/verify_xxx.sh
logs/xxx_test.log
.ai/memory/engineering_issues.md
```

测试报告模板如下：

```markdown
# 测试报告：xxx

## 1. 测试目标
说明本次测试证明什么。

## 2. 测试环境
记录硬件、系统、依赖、设备接口、版本。

## 3. 测试步骤
列出命令、操作、输入数据。

## 4. 预期结果
说明判定通过的标准。

## 5. 实际结果
粘贴关键输出、日志、报文或截图说明。

## 6. 结论
通过 / 不通过 / 部分通过。

## 7. 未覆盖风险
说明未测试到的场景。
```

### 11.5 合并前强制门禁

代码合并或交付前必须满足：

- 编译通过。
- 核心测试通过。
- 新增逻辑有测试或明确的手动验证步骤。
- 重构前后关键功能行为一致。
- 异常输入和错误路径至少覆盖一类。
- 测试结果已记录到文档、日志或最终交付说明中。

不满足以上条件时，不允许标记为完成。

## 12. 薄垂直切片构建策略（Incremental Implementation）

大型遗留系统重构禁止采用大范围横向拆分方式。重构必须优先采用薄垂直切片，让系统始终保持可编译、可运行、可回退。

### 12.1 垂直切片定义

一个薄垂直切片，是指围绕一个最小业务流，从输入到输出完整打通。

示例：单个 CAN 报文解析与展示。

```text
CAN 原始帧
  -> Comm 层接收
  -> Data 层解析
  -> Logic 层状态更新
  -> UI 层显示
  -> 日志记录
  -> 测试验证
```

该切片不要求一次完成全部 CAN 协议、全部 UI 页面、全部存储逻辑，只要求一个最小闭环可运行、可验证。

### 12.2 禁止横向大拆分

禁止以下重构方式：

- 先把所有 UI 文件移动完，再处理逻辑。
- 先把所有通信模块抽离完，再统一测试。
- 一次性重命名大量类、函数、目录。
- 同时修改 UI、Logic、Comm、Data、Config、Utils 多个层级但没有单独验证点。
- 只追求目录结构美观，暂时不保证运行。

### 12.3 推荐切片粒度

每个切片应满足以下条件：

| 要求 | 判定标准 |
|---|---|
| 小 | 只覆盖一个业务入口、一个报文、一个页面区域或一个状态流转 |
| 直 | 从输入到输出能形成完整闭环 |
| 可测 | 有明确命令、日志、UI 操作或测试用例 |
| 可回退 | 修改范围有限，失败后可恢复 |
| 可说明 | 能用一段话说清楚本切片完成了什么 |

### 12.4 标准执行流程

每个切片必须按以下流程执行：

```text
选择一个最小业务流
  -> 写验收标准
  -> 写测试或验证步骤
  -> 实现最小代码
  -> 编译运行
  -> 验证结果
  -> 记录问题
  -> 再进入下一个切片
```

### 12.5 CAN 报文切片示例

以“单个 CAN 报文解析与展示”为例，推荐拆分如下：

| 步骤 | 修改范围 | 验证方式 |
|---|---|---|
| 1 | 定义 `CanFrameModel` 中对应字段 | 单元测试字段默认值 |
| 2 | 在 `CanParser` 中解析该 ID | 输入固定报文，断言解析结果 |
| 3 | 在 `CanDebugController` 中接收解析结果 | mock 一帧报文，检查状态更新 |
| 4 | 在 UI 页面增加字段显示 | 手动发送报文，检查界面显示 |
| 5 | 增加日志和异常报文处理 | 发送非法 DLC / 非法 ID，检查容错 |

完成该切片后，再继续下一个报文或下一个页面区域。

## 13. 安全与防御性编程红线（Security & Hardening）

通信、数据解析、配置加载、日志处理和系统服务属于高风险区域。所有此类代码必须默认输入不可信，必须进行边界检查和异常处理。

### 13.1 通用安全原则

必须遵循以下原则：

- 外部输入一律视为不可信。
- 解析前先检查长度、类型、范围、编码和合法性。
- 异常输入不得导致崩溃、死循环、内存越界或线程阻塞。
- 错误处理应返回明确状态，不允许静默失败。
- 日志必须记录关键错误，但不得泄露密码、密钥、Token 或敏感配置。
- 默认权限应最小化，非必要不使用 root。
- 临时调试开关必须有明确命名、默认关闭，并在文档中说明。

### 13.2 Comm 层红线

Comm 层必须满足：

| 场景 | 强制要求 |
|---|---|
| CAN 报文接收 | 检查 ID、DLC、数据长度、过滤规则 |
| 串口数据接收 | 检查帧头、帧尾、长度、校验、超时 |
| TCP / WebSocket | 检查连接状态、重连间隔、发送队列长度、超时 |
| Modbus | 检查功能码、寄存器范围、CRC、异常响应 |
| ZMQ / 消息队列 | 检查 topic、payload 长度、JSON 字段合法性 |
| ROS topic | 检查 topic 是否存在、频率是否异常、时间戳是否倒退 |

禁止事项：

- 禁止未检查长度就访问数组。
- 禁止无限制缓存接收数据。
- 禁止重连无退避机制。
- 禁止通信线程异常后无日志退出。
- 禁止把设备名、IP、端口写死且不提供配置入口。

### 13.3 Data 层红线

Data 层必须满足：

| 场景 | 强制要求 |
|---|---|
| 二进制协议解析 | 先检查缓冲区长度，再读取字段 |
| JSON 解析 | 检查字段是否存在、类型是否正确、范围是否合法 |
| 单位转换 | 明确单位，例如 `speed_ms`、`speed_kmh` 不得混用 |
| 时间戳处理 | 明确秒、毫秒、纳秒，不允许隐式猜测 |
| 坐标数据 | 明确坐标系、方向、单位、时间基准 |
| 配置数据 | 检查缺省值、非法值和旧版本兼容 |

禁止事项：

- 禁止解析失败后继续使用未初始化数据。
- 禁止用魔法数字解释协议字段。
- 禁止字段单位不写清楚。
- 禁止将异常值直接透传到控制逻辑。
- 禁止对外部输入使用无边界字符串拷贝。

### 13.4 内存与资源管理

C / C++ 项目必须特别遵守：

- new/delete、malloc/free 必须成对管理，优先使用 RAII。
- 文件描述符、socket、串口、CAN 句柄必须在异常路径释放。
- 线程退出必须可控，不允许孤儿线程。
- 定时器、回调、信号槽断开必须有生命周期管理。
- 大对象、大缓存必须有容量上限。
- 禁止在循环中无界增长 vector、queue、map 或日志字符串。

### 13.5 异常报文容错策略

异常通信报文必须按以下策略处理：

```text
接收异常输入
  -> 丢弃当前帧或当前包
  -> 记录有限频率日志
  -> 增加错误计数
  -> 不影响主流程运行
  -> 达到阈值后上报状态
```

禁止因单帧异常报文导致整个服务退出，除非该异常属于明确的安全停机条件。

### 13.6 安全测试要求

涉及 Comm / Data / Config 的修改，必须至少覆盖以下异常输入之一：

- 空输入。
- 长度不足。
- 长度超限。
- 校验错误。
- 非法字段值。
- 未知报文 ID。
- 时间戳异常。
- 配置文件缺失或损坏。
- 连接中断或设备不存在。

## 14. 重构执行步骤建议

重构必须按“基线确认 -> 选择薄垂直切片 -> 测试先行 -> 最小实现 -> 验证 -> 记录 -> 下一个切片”的方式推进。禁止为了追求目录美观而先做大范围横向拆分。

### 14.1 阶段一：基线确认

1. 确认当前代码能否编译、启动和运行。
2. 记录当前功能入口、核心流程、通信接口、配置文件和日志位置。
3. 保存当前可运行状态的测试命令、运行日志或截图。
4. 明确本次不修改的功能范围。

### 14.2 阶段二：选择第一个薄垂直切片

1. 只选择一个最小业务流。
2. 明确该业务流的输入、处理、输出和验收标准。
3. 明确涉及文件，不允许顺手修改无关模块。
4. 示例：只处理一个 CAN ID 的解析、状态更新和 UI 显示。

### 14.3 阶段三：测试先行

1. 为该切片写单元测试、脚本测试或手动验证步骤。
2. Logic / Comm / Data 层修改应优先写失败测试。
3. 无硬件环境时，使用 mock、vcan、回环、样例日志或离线数据验证。
4. 不允许在没有验证方法的情况下进入生产代码修改。

### 14.4 阶段四：最小实现

1. 只写让当前切片通过的最小代码。
2. 不重写无关模块。
3. 不新增万能类。
4. 不把业务逻辑塞入 UI。
5. 不将临时调试逻辑混入正式代码。

### 14.5 阶段五：验证与回归

1. 编译测试。
2. 当前切片功能测试。
3. 异常输入测试。
4. 原有关键功能回归测试。
5. 日志、状态、错误提示检查。

### 14.6 阶段六：记录与交付

1. 记录修改文件。
2. 记录测试命令和实际结果。
3. 记录发现的问题、原因和处理方法。
4. 更新模块文档和 Doxygen 注释。
5. 明确回退方法。
6. 当前切片通过后，才能进入下一个切片。

## 15. 页面与模块对应关系建议

推荐采用“页面—控制器—底层模块”的组织方式：

```text
UI页面                Logic控制器                 底层模块
---------------------------------------------------------------
CanDebugPage      ->  CanDebugController     ->  CanDriver
                                           ->  CanParser
                                           ->  CanRecordStorage

SerialPage        ->  SerialController       ->  SerialDriver
                                           ->  SerialParser

MonitorPage       ->  MonitorManager         ->  DeviceStatusModel
                                           ->  DataCache

ConfigPage        ->  ConfigController       ->  ConfigManager
```

这样做的优点：

- 页面职责清晰
- 控制流程清晰
- 底层功能可替换
- 出问题时便于快速定位

## 16. 必须避免的问题

### 16.1 UI文件过重
禁止在 `mainwindow.cpp` 或某个页面 cpp 中堆积：

- 通信逻辑
- 协议解析
- 数据存储
- 大量业务流程判断
- 大段重复代码

### 16.2 万能类
禁止一个类同时负责：

- UI显示
- 数据处理
- 通信发送
- 配置读取
- 日志写入

这类代码后期最难维护。

### 16.3 功能散落
同一个功能相关代码不要散落在大量不相关文件中，应围绕模块统一组织。

### 16.4 核心代码无说明
以下内容禁止缺失说明：

- 协议字段处理
- 状态机
- 控制流程
- 异常处理
- 特殊兼容逻辑
- 关键时序逻辑

---

## 17. 最终验收标准

项目整改完成后，应满足以下标准：

### 17.1 结构方面
- 目录清晰
- 分层明确
- 文件分类统一
- 模块职责单一

### 17.2 UI方面
- 功能入口齐全
- 参数配置明确
- 状态显示清楚
- 数据展示合理
- 日志调试方便
- 页面分类清晰

### 17.3 代码方面
- UI与业务逻辑分离
- 逻辑与通信分离
- 数据处理独立
- 配置统一管理
- 公共方法可复用
- 注释风格统一

### 17.4 文档方面
- 每个核心模块有说明
- 每个核心类有注释
- 每个主要函数有 Doxygen 注释
- 模块调用关系可追踪
- 新接手人员可快速理解项目结构

### 17.5 测试方面
- 关键 Logic / Comm / Data 修改有测试或手动验证记录
- 重构前后行为有对比
- 异常输入和边界场景有覆盖
- 测试报告、日志或验证命令已归档
- 未覆盖风险已明确说明

### 17.6 安全方面
- 外部输入有长度、范围、类型检查
- 异常通信报文不会导致程序崩溃
- 资源释放路径完整
- 无无界缓存、无无限重连、无敏感信息日志泄露
- 默认权限最小化

## 18. 长期执行规则

后续开发中，建议长期固定执行以下规则：

### 规则1
凡是能在界面中展示的功能，尽量放到界面中。

### 规则2
UI层只做显示与交互，不写复杂业务逻辑。

### 规则3
每新增一个功能，尽量独立成模块和文件。

### 规则4
通信、数据、配置、逻辑、工具必须分层。

### 规则5
每个主要文件、类、函数必须使用 Doxygen 风格注释。

### 规则6
每个核心模块必须提供对应说明文档。

### 规则7
结构整理、模块拆分、UI可视化、注释规范要作为长期标准执行，而不是一次性整改。

### 规则8
Logic、Comm、Data、Config 等核心层修改必须有测试或验证记录；没有证明，不算完成。

### 规则9
大型重构必须按薄垂直切片推进，禁止一次性横向大改。

### 规则10
通信和数据解析必须默认外部输入不可信，必须进行边界检查和异常容错。

### 规则11
任何人或 AI 都不得以“简单、临时、赶时间、稍后补”为理由绕过本规范。

## 19. AI 编码助手协作规范

本章节参考 `addyosmani/agent-skills` 的工作流思想，将 AI 编码助手从“直接写代码”约束为“按阶段执行、按门禁验证、按文档交付”的工程化协作方式。

### 19.1 核心原则

AI 编码助手不得默认直接实现代码。所有较大的新增功能、重构、修复、接口调整、通信协议变更、UI 调整、系统服务修改，都应先进入规范流程。

推荐生命周期如下：

```text
DEFINE  ->  PLAN  ->  BUILD  ->  TEST  ->  REVIEW  ->  SHIP
需求定义    任务拆分    增量实现    测试验证    代码审查    交付归档
```

对应执行原则：

| 阶段 | 目标 | 主要产物 | 是否允许跳过 |
|---|---|---|---|
| DEFINE | 明确要做什么、不做什么、验收标准是什么 | 需求说明 / 规格说明 | 不允许 |
| PLAN | 拆成小任务，明确顺序和依赖 | 开发计划 / 任务清单 | 不允许 |
| BUILD | 按小切片逐步实现 | 代码变更 | 不允许一次性大改 |
| TEST | 证明功能确实可用 | 测试记录 / 命令输出 / 日志 | 不允许 |
| REVIEW | 检查结构、风险、可维护性 | Review 结论 / 问题清单 | 不允许 |
| SHIP | 形成可交付状态 | 变更说明 / 使用说明 / 回退方案 | 不允许 |

### 19.2 AI 执行入口映射

后续使用 Codex、Cursor、Copilot、Claude Code、OpenCode、Gemini CLI 或其他 AI 编码工具时，应按任务类型自动选择执行流程。

| 用户意图 | 必须进入的流程 | 说明 |
|---|---|---|
| 新功能开发 | Spec -> Plan -> Build -> Test -> Review | 先写方案，确认后再改代码 |
| 项目重构 | Spec -> Plan -> Incremental Build -> Regression Test -> Review | 必须保证原功能不破坏 |
| Bug 修复 | Reproduce -> Root Cause -> Fix Plan -> Build -> Verify | 先复现和定位，不允许盲改 |
| UI 修改 | UI Spec -> Layout Plan -> Build -> Visual Check -> Review | 所有入口、状态、参数尽量可视化 |
| 通信协议修改 | Protocol Spec -> Compatibility Check -> Build -> Packet Test | 必须说明字段、帧格式、兼容影响 |
| 系统服务修改 | Service Spec -> Risk Check -> Build -> Boot/Runtime Test | 必须包含启动、停止、日志、回退 |
| 性能优化 | Baseline -> Hypothesis -> Change -> Measurement -> Compare | 必须有优化前后对比 |
| 文档整理 | Source Review -> Structure Plan -> Write -> Consistency Check | 文档必须能指导后续执行 |

### 19.3 AI 不允许的行为

以下行为必须禁止：

1. 未写方案直接改核心代码。
2. 未确认边界就扩大改动范围。
3. 为了快速完成而跳过测试。
4. 只说明“应该可以”，但不给验证命令和结果。
5. 把多个无关修改混在一次提交里。
6. 未说明风险、回退方案、影响范围。
7. 将临时调试代码混入正式逻辑。
8. 为了兼容某个现象写死路径、IP、端口、设备名，但不说明原因。
9. 未经确认删除原有功能、原有配置或原有脚本。
10. 遇到报错后反复试错，不记录问题原因和处理结果。

### 19.4 AI 必须输出的内容

每次涉及代码或项目结构变更时，AI 至少需要输出以下内容：

```text
1. 修改目标
2. 当前现状
3. 问题或需求分析
4. 修改方案
5. 涉及文件
6. 风险点
7. 验证方法
8. 回退方法
9. 是否需要用户确认后再执行
```

对于较大任务，必须先生成 `docs/xxx_plan.md` 或 `docs/xxx_spec.md`，经确认后再写代码。

### 19.5 最小可验证切片原则

开发不得一次性完成大量修改。应按最小可验证切片执行。

示例：

```text
错误方式：
一次性重构 UI、通信、数据解析、日志、配置保存。

正确方式：
1. 先整理目录结构，不改逻辑。
2. 再抽离通信模块，验证通信功能。
3. 再抽离数据解析，验证解析结果。
4. 再整理 UI 展示，验证界面入口。
5. 最后补文档和注释。
```

每一个切片都必须满足：

- 改动范围明确
- 可单独编译或运行
- 有验证命令
- 有预期结果
- 出错可回退

### 19.6 质量门禁

每个阶段完成后必须经过门禁检查。

#### 16.6.1 方案门禁

在进入编码前，必须确认：

- 需求目标明确
- 不做事项明确
- 输入输出明确
- 涉及模块明确
- 验收标准明确
- 风险点明确

#### 16.6.2 编码门禁

代码提交前必须确认：

- 是否符合目录分层
- UI 是否只负责显示和交互
- Logic 是否负责业务调度
- Comm 是否独立于 UI
- Data 是否独立负责模型、解析、转换、存储
- Config 是否统一管理配置
- Utils 是否没有混入业务逻辑
- 核心文件、类、函数是否有 Doxygen 注释

#### 16.6.3 测试门禁

测试必须包含：

- 编译测试
- 启动测试
- 核心功能测试
- 异常输入测试
- 日志检查
- 回归测试

对嵌入式、通信、车辆、CAN、ROS、系统服务类项目，还应补充：

- 设备节点检查
- 权限检查
- 接口状态检查
- 服务状态检查
- 报文收发检查
- 时间戳检查
- 开机自启动检查

#### 16.6.4 Review 门禁

Review 必须检查：

- 是否存在万能类
- 是否存在 UI 重逻辑
- 是否存在重复代码
- 是否存在硬编码
- 是否存在隐藏控制流
- 是否存在未处理异常
- 是否存在无说明的兼容逻辑
- 是否存在无法回退的修改
- 是否存在破坏原功能的风险

### 19.10 上下文加载规则

AI 上下文不能一次性塞入所有规则和所有文档。应按任务加载相关内容。

推荐规则：

| 当前任务 | 必须加载 |
|---|---|
| 新功能开发 | 总规范 + spec + plan + 当前模块说明 |
| Bug 修复 | 总规范 + 问题日志 + 相关模块代码 + 测试方法 |
| UI 调整 | 总规范 + UI 规范 + 页面代码 |
| CAN 调试 | 总规范 + CAN 协议说明 + comm/can + data/parser |
| ROS 调试 | 总规范 + ROS topic 说明 + launch + 日志 |
| Buildroot 开发 | 总规范 + package 说明 + board overlay + 构建日志 |

禁止一次性加载大量无关文件，避免 AI 忽略关键上下文。

### 19.11 问题记录与长期记忆文件

项目必须维护长期问题记录文件：

```text

```

每次遇到问题都必须追加记录：

```markdown
## YYYY-MM-DD 问题标题

### 现象
记录终端输出、日志、错误信息。

### 原因
说明根因，不确定时写“暂未完全确认”。

### 解决方法
记录实际有效命令或代码修改。

### 验证结果
记录验证命令和输出结果。

说明以后如何避免重复发生。

### 19.12 AI 交付格式

AI 完成任务后，最终回复必须包含：

```text
1. 已完成内容
2. 修改文件
3. 关键逻辑
4. 验证方法
5. 风险与限制
6. 后续建议
```

如果未能完成，必须明确说明：

```text
1. 已完成到哪一步
2. 卡在哪里
3. 缺少什么信息
4. 下一步应该怎么做
```

不得用“应该可以”“大概没问题”作为结论。

## 20. AGENTS.md 建议模板

可以在项目根目录新增 `AGENTS.md`，内容如下：

```markdown
# AGENTS.md

本文件用于约束 AI 编码助手在本项目中的行为。

## 1. 项目基本规则

- 先写方案，再写代码。
- 方案未经确认，不允许直接改核心代码。
- 使用最小代码量完成目标。
- 保持目录结构清晰。
- 不允许扩大需求范围。
- 不允许删除原有功能，除非用户明确要求。
- 不允许隐藏控制流。
- 不允许把临时调试逻辑混入正式代码。
- 遇到问题必须记录到 `.ai/memory/engineering_issues.md`。

## 2. 必须遵守的开发流程

```text
DEFINE -> PLAN -> BUILD -> TEST -> REVIEW -> SHIP
```

## 3. 分层规则

- UI 层只负责显示和交互。
- Logic 层负责业务流程控制。
- Comm 层负责通信。
- Data 层负责模型、解析、转换、缓存、存储。
- Config 层负责配置加载、保存、默认值。
- Utils 层只放通用工具，不放业务逻辑。

## 4. 修改前必须输出

- 修改目标
- 当前问题
- 修改方案
- 涉及文件
- 风险点
- 验证方法
- 回退方案

## 5. 编码要求

- 每个主要文件、类、函数必须有 Doxygen 注释。
- 一个模块只负责一类事情。
- 新增功能优先独立成文件。
- 禁止万能类。
- 禁止 UI 文件堆积业务逻辑。
- 禁止无说明硬编码。

## 6. 测试要求

每次修改后至少提供：

- 编译命令
- 运行命令
- 功能验证命令
- 日志检查命令
- 预期结果

## 7. 输出要求

最终回复必须说明：

- 已完成内容
- 修改文件
- 验证结果
- 风险限制
- 下一步建议
```

## 21. AI 代码审查检查表

每次 AI 修改代码后，必须按以下表格自查。

| 检查项 | 通过标准 | 结果 |
|---|---|---|
| 需求边界 | 未超出用户要求 | 待填写 |
| 目录结构 | 符合 ui/logic/comm/data/config/utils 分层 | 待填写 |
| UI 逻辑 | UI 未直接处理复杂业务和协议 | 待填写 |
| 通信模块 | 通信代码不依赖 UI | 待填写 |
| 数据模块 | 数据模型、解析、转换独立 | 待填写 |
| 配置模块 | 配置可加载、可保存、有默认值 | 待填写 |
| 注释 | 核心文件、类、函数具备 Doxygen 注释 | 待填写 |
| 测试 | 有明确验证命令和结果 | 待填写 |
| 日志 | 关键异常和状态有日志 | 待填写 |
| 回退 | 有恢复旧版本或关闭新功能的方法 | 待填写 |

## 22. 反合理化防逃避机制（Anti-Rationalization）

本章节用于约束人工开发者和 AI 编码助手在执行规范时可能出现的走捷径行为。凡是落入下表的理由，一律不得作为违反规范的依据。

| 常见借口 | 明确禁止的行为 | 正确做法 |
|---|---|---|
| 代码逻辑很简单 | 跳过 Doxygen 注释 | 核心文件、类、函数仍必须写清楚用途、输入输出和限制 |
| 只是临时调试 | 把临时开关、打印、硬编码混入正式逻辑 | 使用明确命名的 debug 开关，默认关闭，并记录用途 |
| 为了提高效率 | 让 UI 层直接读取底层数据或协议字段 | UI 只调用 Logic 层接口，由 Logic 调度 Data / Comm |
| 后面再补测试 | 先合并代码，推迟测试文档 | 修改完成前必须保留测试命令、日志或测试报告 |
| 当前没有硬件 | 完全跳过验证 | 使用 mock、vcan、回环、样例数据或手动验证步骤替代 |
| 这次只改一点点 | 不写方案、不列影响范围 | 至少说明修改目标、涉及文件、验证方法和回退方式 |
| 旧代码本来就乱 | 继续增加万能类或把逻辑堆进 mainwindow.cpp | 通过薄垂直切片逐步隔离，不能继续扩大混乱 |
| 赶进度 | 跳过边界检查、异常处理、日志记录 | 可以缩小切片范围，但不能降低安全和验证标准 |
| 只是内部工具 | 不做输入校验、不处理异常 | 内部工具同样可能连接真实设备，必须具备最低防御能力 |
| AI 生成的代码看起来能跑 | 不审查、不测试直接使用 | 必须经过 Review、编译、运行和关键路径验证 |
| 目录先整理，功能后面再修 | 大规模移动文件后系统不可运行 | 每次移动都必须保持可编译、可运行、可回退 |
| 日志太麻烦 | 不记录异常和关键状态 | 至少记录错误原因、关键输入、处理结果和错误计数 |
| 这个字段大家都知道 | 不写单位、范围、坐标系或协议含义 | 字段定义必须写明单位、范围、来源和使用位置 |
| 配置先写死 | 将路径、IP、端口、设备名散落在代码中 | 进入 Config 层或集中常量，并说明默认值和覆盖方式 |

### 22.1 违反规范时的处理原则

发现违反规范的代码时，不允许继续在其基础上叠加新功能。必须先做以下处理：

1. 标记违反项。
2. 说明风险。
3. 给出最小修复方案。
4. 补充验证方法。
5. 修复后再继续功能开发。

### 22.2 AI 自检要求

AI 在输出代码、方案或修改建议前，必须自查是否存在以下问题：

- 是否跳过了测试或验证。
- 是否扩大了用户未要求的范围。
- 是否让 UI 越权访问 Comm / Data。
- 是否未说明风险和回退方案。
- 是否未记录异常和问题原因。
- 是否用“简单”“临时”“稍后补”作为省略理由。

只要存在任一问题，AI 必须先修正输出，再交付。

## 23. 最终结论补充

本规范不仅约束人工开发，也约束 AI 编码助手参与开发时的行为。

后续项目开发应固定执行：

1. 先写规格说明。
2. 再拆分任务。
3. 再按薄垂直切片实现。
4. 每一步都要能验证，测试即证明。
5. 通信和数据解析必须具备边界防御。
6. 修改后必须 Review。
7. 交付时必须有说明、日志、测试和回退方法。
8. 禁止使用任何借口绕过注释、测试、分层、安全和文档要求。

这样可以避免 AI 直接生成大量不可控代码，也能保证项目长期具备可维护性、可调试性和可交接性。

## 24. 结论

本规范适用于项目重构、功能新增、后续维护、交接文档编写和代码评审。

后续无论是进行 UI 项目开发，还是通信、控制、测试、诊断类功能开发，都应优先遵循本规范，以保证项目具备以下特性：

- 结构清晰
- 功能清楚
- 便于调试
- 便于维护
- 便于扩展
- 便于演示
- 便于交接

如后续项目中存在特殊模块，也应在不破坏本规范总体原则的前提下进行局部扩展。
